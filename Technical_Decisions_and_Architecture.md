# تصمیمات فنی و معماری پیشنهادی پروژه

## 1. انتخاب تکنولوژی پایه
پلتفرم اصلی پروژه ASP.NET Core انتخاب می‌شود. این انتخاب به‌دلیل پایداری، مقیاس‌پذیری، امنیت بالا و تجربه توسعه‌دهنده انجام شده است.

---

## 2. انتخاب MVC به‌جای Razor Pages
Razor Pages برای پروژه‌های ساده و فرم‌محور مناسب است. با توجه به ماهیت داشبوردی، تعاملی و توسعه‌پذیر این پروژه، ASP.NET Core MVC انتخاب می‌شود تا انعطاف بیشتری در مدیریت UI و منطق داشته باشیم.

---

## 3. معماری Hybrid (MVC + API)
پروژه به‌صورت Hybrid طراحی می‌شود:
- MVC برای رندر صفحات
- Web API برای منطق تعاملی و داده‌ها

این API در ابتدا داخلی است اما قابلیت عمومی شدن دارد.

---

## 4. ضرورت وجود API حتی در فاز اول
وجود API باعث:
- جداسازی منطق از UI
- تست‌پذیری بهتر
- آماده‌سازی برای موبایل و SPA
- استفاده مجدد از منطق کسب‌وکار

می‌شود.

---

## 5. عدم نیاز به اپ موبایل در فاز اول
در فاز اول تمرکز فقط روی Web App است. با این حال، معماری از ابتدا Mobile-Ready طراحی می‌شود.

---

## 6. مسیر صحیح توسعه موبایل
مسیر پیشنهادی:
1. Web App
2. PWA
3. Mobile App (در صورت نیاز)

این مسیر کم‌ریسک و کم‌هزینه است.

---

## 7. انتخاب Front-End سبک
در فاز MVP از تکنولوژی‌های زیر استفاده می‌شود:
- Razor View
- jQuery
- Vanilla JS یا Alpine.js
- Chart.js
- FullCalendar

بدون SPA سنگین.

---

## 8. لایه‌بندی پروژه
پروژه شامل لایه‌های زیر خواهد بود:
- Presentation
- Application
- Domain
- Infrastructure

این ساختار توسعه‌پذیر و قابل نگهداری است.

---

## 9. دیتابیس و ORM
- SQL Server
- EF Core
- Soft Delete
- Audit Fields

طراحی دیتابیس مستقل از API اینستاگرام انجام می‌شود.

---

## 10. سیاست تعامل با اینستاگرام
- عدم وابستگی بحرانی به API رسمی
- امکان ورود دستی داده‌ها
- آماده‌سازی برای اتصال محدود در آینده

---

## جمع‌بندی
این تصمیمات فنی تضمین می‌کنند که پروژه:
- سریع راه‌اندازی شود
- مقیاس‌پذیر باشد
- هزینه توسعه کنترل شود
- برای آینده آماده بماند
